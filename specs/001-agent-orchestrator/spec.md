# Feature Specification: Autonomous Agent Orchestrator Platform

**Feature Branch**: `001-agent-orchestrator`
**Created**: 2026-03-07
**Status**: Draft
**Input**: User description: "I want to build an always on 24/7 agent that runs in the cloud / not on my local machine. I want it to be super secure, and be able to work on parallel work in entirely different domain spaces. Essentially this root agent should be an orchestrator agent that can spin out specialized agents. Some use cases I have in mind - 1. A team of agents that scrape the web for app business ideas, do the research, build the app, deploy / host the app, market the app, and reinvest the proceeds into themselves (think self growing). I want agents to have full computer usage so they can do truly anything a human could if they had a computer. They could go out and create their own phone numbers, emails, buy themselves voices, set up accounts with different tools with said emails, etc."

## User Scenarios & Testing

### User Story 1 - Deploy and Start the Orchestrator (Priority: P1)

The operator deploys the orchestrator platform to a cloud environment and starts it. The orchestrator boots up, establishes secure communication channels, and begins listening for work assignments or self-directed goal evaluation. The operator can verify the orchestrator is running, healthy, and secured via a management dashboard.

**Why this priority**: Nothing else works without a running, secure, always-on orchestrator. This is the foundation for all other capabilities.

**Independent Test**: Can be fully tested by deploying the orchestrator, verifying it starts, confirming health checks pass, and accessing the management dashboard to see system status.

**Acceptance Scenarios**:

1. **Given** the orchestrator is not running, **When** the operator deploys it to the cloud environment, **Then** the orchestrator starts within 5 minutes, passes all health checks, and reports "healthy" status on the management dashboard.
2. **Given** the orchestrator is running, **When** the operator views the management dashboard, **Then** they see system health, uptime, resource usage, and the number of active/idle specialized agents.
3. **Given** the orchestrator is running, **When** a hardware or process failure occurs, **Then** the orchestrator automatically restarts and resumes its previous state within 2 minutes.

---

### User Story 2 - Spawn and Manage Specialized Agents (Priority: P1)

The orchestrator receives a goal (either from the operator or self-generated) and spawns one or more specialized agents to accomplish it. Each specialized agent runs in its own isolated environment with full computer usage capabilities (browser, terminal, file system). The orchestrator tracks agent progress, can pause/resume/terminate agents, and reclaims resources when agents complete their work.

**Why this priority**: The core value proposition is the ability to dynamically create and manage specialized agents. Without this, the platform is just a single process.

**Independent Test**: Can be tested by submitting a goal to the orchestrator, verifying a specialized agent is spawned in an isolated environment, monitoring its progress, and confirming it can be terminated cleanly.

**Acceptance Scenarios**:

1. **Given** the orchestrator is running and idle, **When** a goal is submitted (e.g., "research trending SaaS ideas"), **Then** the orchestrator spawns a specialized agent within 30 seconds with the appropriate capabilities and an isolated execution environment.
2. **Given** a specialized agent is running, **When** the operator requests termination, **Then** the agent is gracefully shut down, its resources are reclaimed, and its output/state is preserved.
3. **Given** multiple goals are submitted across different domains, **When** the orchestrator processes them, **Then** it spawns separate agents for each goal running in parallel without interference.
4. **Given** a specialized agent crashes, **When** the orchestrator detects the failure, **Then** it logs the failure, preserves any partial output, and optionally restarts the agent based on configured retry policy.

---

### User Story 3 - Full Computer Usage for Agents (Priority: P1)

Each specialized agent has full computer usage capabilities equivalent to a human sitting at a computer. This includes browsing the web, interacting with web applications, using command-line tools, managing files, sending/receiving emails, making API calls, and interacting with any software that runs on a computer. Agents can create accounts on services, set up phone numbers, purchase services, and operate autonomously.

**Why this priority**: The differentiating capability of this platform is that agents can do anything a human can do with a computer. This is essential for the self-growing use case.

**Independent Test**: Can be tested by instructing an agent to perform a multi-step task requiring browser interaction, command-line usage, and account creation on a test service.

**Acceptance Scenarios**:

1. **Given** a specialized agent is running, **When** it is instructed to browse a website and extract information, **Then** it successfully navigates the site, interacts with page elements, and returns structured data.
2. **Given** a specialized agent is running, **When** it is instructed to create an email account on a supported provider, **Then** it completes the registration flow, stores the credentials securely, and confirms the account is active.
3. **Given** a specialized agent is running, **When** it needs to use a command-line tool, **Then** it can install, configure, and execute CLI tools in its isolated environment.
4. **Given** a specialized agent is running, **When** it needs to make a purchase or financial transaction, **Then** it uses operator-approved payment methods with configurable spending limits and logs all transactions.

### User Story 4 - Security and Isolation (Priority: P1)

All agents run in fully isolated environments where they cannot access each other's data, credentials, or resources unless explicitly permitted. The orchestrator enforces authentication, authorization, and encryption for all communications. Secrets (API keys, credentials, payment methods) are stored in a dedicated vault and only injected into agent environments on a need-to-know basis. All agent actions are logged in an immutable audit trail.

**Why this priority**: Security is a hard requirement stated by the operator. Agents with full computer usage operating autonomously require strong isolation and controls to prevent unauthorized actions and data leaks.

**Independent Test**: Can be tested by spawning two agents and verifying neither can access the other's environment, credentials, or data. Verify all communications are encrypted and audit logs capture all actions.

**Acceptance Scenarios**:

1. **Given** two specialized agents are running, **When** Agent A attempts to access Agent B's file system or credentials, **Then** the attempt is denied and logged as a security event.
2. **Given** a specialized agent needs an API key, **When** it requests the secret from the vault, **Then** the orchestrator verifies the agent is authorized for that secret before injecting it, and the secret is never written to disk unencrypted.
3. **Given** any agent performs an action, **When** the action completes, **Then** an immutable audit log entry is created containing the agent ID, action type, timestamp, and outcome.
4. **Given** an agent attempts an action outside its authorized scope, **When** the orchestrator detects the violation, **Then** the action is blocked, the operator is notified, and the event is logged.

### User Story 5 - Self-Growing Business Pipeline (Priority: P2)

The operator configures a self-growing pipeline where agents autonomously: (1) research and identify promising app/business ideas, (2) validate the ideas through market research, (3) build the application, (4) deploy and host the application, (5) market the application, (6) collect revenue, and (7) reinvest proceeds into expanding agent capabilities or launching new ventures. The operator sets guardrails (budget limits, domain restrictions, approval gates) and monitors progress through the dashboard.

**Why this priority**: This is the primary use case described by the operator, but it depends on all P1 stories being functional first. It is a composition of the foundational capabilities.

**Independent Test**: Can be tested by configuring a pipeline with a small budget, verifying agents progress through each stage (research, build, deploy, market), and confirming financial transactions are logged and within limits.

**Acceptance Scenarios**:

1. **Given** a self-growing pipeline is configured with a $100 budget, **When** agents begin the research phase, **Then** they produce a ranked list of at least 5 business ideas with market size estimates and feasibility scores.
2. **Given** agents have selected a business idea, **When** they enter the build phase, **Then** they produce a working application with source code, documentation, and deployment configuration.
3. **Given** an application is built, **When** agents enter the deployment phase, **Then** the application is deployed to a hosting platform, is publicly accessible, and passes basic functionality checks.
4. **Given** an application is deployed, **When** agents enter the marketing phase, **Then** they create marketing materials and execute at least one marketing channel (e.g., social media, SEO, paid ads within budget).
5. **Given** revenue is generated, **When** the reinvestment threshold is met, **Then** agents allocate funds according to the operator's reinvestment rules and log all financial decisions.

### User Story 6 - Operator Dashboard and Controls (Priority: P2)

The operator accesses a management dashboard as the control plane for configuring agents, building agent definitions, setting budgets and spending limits, configuring approval gates, and monitoring system health. Live interaction with running agents (approvals, questions, status updates, commands) happens through Discord as two-way conversational messaging — agents communicate like real people in chat channels. The operator can also pause, resume, redirect, or terminate agents from either the dashboard or Discord commands.

**Why this priority**: Operators need visibility and control over autonomous agents, especially when agents handle finances and create external accounts. This is critical for trust and safety.

**Independent Test**: Can be tested by accessing the dashboard, viewing active agents, triggering a pause/resume on a running agent, and configuring a spending limit that agents respect.

**Acceptance Scenarios**:

1. **Given** the dashboard is loaded, **When** agents are running, **Then** the operator sees a real-time list of all agents with their status, current task, resource usage, and elapsed time.
2. **Given** a spending limit of $50 is configured, **When** an agent attempts a transaction exceeding the limit, **Then** the transaction is blocked and the operator is notified for approval.
3. **Given** an approval gate is configured for account creation, **When** an agent attempts to create a new external account, **Then** the action is paused pending operator approval via the dashboard.

### User Story 7 - Agent Identity Management (Priority: P3)

Agents can create and manage their own digital identities: email addresses, phone numbers, voice profiles, and accounts on third-party services. These identities are tracked in a central registry, associated with the agent that created them, and can be transferred between agents or decommissioned. The operator has full visibility into all identities and can revoke any identity at any time.

**Why this priority**: Enables agents to fully operate autonomously in the real world, but depends on full computer usage (P1) and security (P1) being in place first.

**Independent Test**: Can be tested by instructing an agent to create an email address and phone number, verifying they appear in the identity registry, and confirming the operator can revoke them.

**Acceptance Scenarios**:

1. **Given** an agent needs an email identity, **When** it creates one through a supported provider, **Then** the email address, credentials, and associated agent ID are stored in the identity registry.
2. **Given** an agent has created identities, **When** the operator views the identity registry, **Then** all identities are listed with their type, creation date, associated agent, and active/revoked status.
3. **Given** an identity is no longer needed, **When** the operator revokes it, **Then** the identity is deactivated on the provider (where possible), marked as revoked in the registry, and no agent can use it further.

### Edge Cases

- What happens when the orchestrator runs out of compute resources while spawning new agents? → Resolved: priority queue suspends lower-priority agents; resource constraints surfaced as orchestrator signals (see FR-015a).
- How does the system handle an agent that becomes unresponsive or enters an infinite loop? → Resolved: Heartbeat + resource heuristics + periodic LLM self-assessment. Heartbeats detect hangs, CPU/memory/LLM-cost spike detection catches infinite loops, and periodic LLM progress checks catch subtle stalls. Configurable thresholds and kill/restart policies per agent (see FR-012a, FR-015b).
- What happens when an external service (e.g., email provider) blocks or rate-limits agent activity? → Resolved: Agent-autonomous resolution first. Agents decide their own retry/workaround strategy (backoff, rotate identity/proxy, try alternative services). Agents self-learn from failures — resolved incidents are internalized as learned context to expedite future similar issues. Operator escalation is a true last resort only after agent-autonomous strategies are exhausted (see FR-012b, FR-012c).
- How does the system recover if the secrets vault becomes temporarily unavailable? → Resolved: Graceful degradation. Agents continue non-sensitive work with cached credentials, queue sensitive operations, and self-heal when vault recovers (see FR-015c).
- What happens when two agents inadvertently compete for the same external resource (e.g., same domain name)? → Resolved: Parent-mediated conflict resolution. Agents detect contention and escalate to their shared parent agent via the message queue. The parent (LLM-powered) evaluates priorities, context, and constraints to resolve the conflict — reassigning resources, choosing a winner, or redirecting the losing agent to an alternative. Cross-workflow conflicts escalate up the hierarchy to the root orchestrator. Resolved conflicts are published to the shared knowledge base to prevent recurrence.
- How does the system handle network partitions between the orchestrator and running agents? → Resolved: Graceful degradation — agents continue autonomous work with cached state, queue orchestrator-dependent operations, and reconcile when connectivity restores.
- What happens when an agent's spending approaches or exceeds configured budget limits mid-transaction? → Resolved: Pre-authorization check. Agents must verify sufficient budget before initiating any transaction; transaction is blocked if insufficient funds remain. See FR-008c.
- How does the system handle credential rotation for long-running agents? → Resolved: Graceful degradation — agents use cached credentials until expiry, request rotation from vault when available, and queue credential-dependent work if vault is unreachable.

## Requirements

### Functional Requirements

- **FR-001**: System MUST provide a root orchestrator process that runs continuously (24/7) in a cloud environment and automatically recovers from failures.
- **FR-002**: System MUST allow any agent (orchestrator or specialized agent) to spawn sub-agents on demand, each running in an isolated execution environment, creating a hierarchical agent tree. Each parent agent acts as a mini-orchestrator for its children — delegating tasks, resteering, resolving conflicts between children, and collecting results. Budget, permissions, and capability constraints cascade downward from parent to child. The root orchestrator sits at the top of the hierarchy.
- **FR-003**: System MUST provide each specialized agent with full computer usage capabilities including web browsing, terminal access, file system access, and the ability to interact with any web-based or desktop application. Agents MUST follow a tiered interaction hierarchy: prefer direct API/SDK calls first, headless browser automation (Playwright-style) second, and full pixel-level screen/mouse/keyboard computer use only as a last resort for tasks requiring visual UI interaction.
- **FR-004**: System MUST enforce strict isolation between agent environments such that no agent can access another agent's data, credentials, or processes. All inter-agent data handoff MUST flow through the orchestrator. The orchestrator uses the E2B SDK to inject initial task context (structured prompt files, environment variables, secrets) at agent spawn and reads final result artifacts from the sandbox filesystem on completion. During execution, agents communicate with the orchestrator via Redis with Streams (using BullMQ for job queues) for mid-task progress updates, decision requests, escalations, and resteering commands. No direct agent-to-agent communication is permitted — all messages route through the orchestrator or parent agent.
- **FR-005**: System MUST encrypt all inter-agent and orchestrator-agent communications in transit and at rest.
- **FR-006**: System MUST provide a secrets vault for storing API keys, credentials, and payment methods, with access controlled on a per-agent, per-secret basis. Secrets MUST be delivered to agents exclusively through a secure API proxy (FR-026), never as raw environment variables or plaintext files.
- **FR-007**: System MUST maintain an immutable audit log of all agent actions, including timestamps, agent IDs, action types, and outcomes.
- **FR-008**: System MUST support configurable spending limits and guardrails per agent. Agents act freely within their pre-set budgets and rules without requiring approval. When an agent encounters an action that exceeds its configured limits or authority, it MUST escalate to the operator via Discord and pause until approved.
- **FR-008a**: System MUST enforce a configurable revenue split (default: 20% operator / 80% agent reinvestment). Agents retain their share as a reinvestment budget and MUST only spend when it demonstrably enables higher revenue or advances the agent's defined north star metric. All spending decisions MUST be logged with justification tied to the agent's mission/guidance.
- **FR-008c**: Agents MUST perform a pre-authorization budget check before initiating any financial transaction. If the transaction amount would exceed the agent's remaining budget, the transaction MUST be blocked before execution. The agent MAY escalate to the operator to request additional budget.
- **FR-008b**: System MUST track LLM API costs at per-agent, per-model granularity — recording input tokens, output tokens, cache reads, and computed cost for every LLM call. Total agent cost (LLM + compute + storage + external services) informs the orchestrator's resource allocation and reinvestment decisions.
- **FR-010b**: The management dashboard MUST display an interactive agent hierarchy tree visualizing the full parent→child agent structure in real-time. Each node MUST show: agent name/ID, status (live/idle/error/suspended), current task summary, budget usage (spent/remaining), and model in use. Subtrees MUST be collapsible for large hierarchies. Status changes MUST propagate via SSE (FR-010).
- **FR-008d**: The management dashboard MUST surface token economics: per-agent per-model cost breakdown, cost trend charts (7d/30d toggles), projected monthly spend, and per-venture ROI tracking (revenue vs total operating cost). Smart alert banners MUST surface automatically when costs exceed configurable thresholds or spending velocity indicates budget exhaustion before period end.
- **FR-009**: System MUST allow agents to create and manage digital identities (email addresses, phone numbers, service accounts) and track them in a central identity registry.
- **FR-010**: System MUST provide a management dashboard as the control plane for agent configuration, agent definition building, budget/gate settings, and system health monitoring. The dashboard MUST use Server-Sent Events (SSE) for real-time push updates — agent status changes, cost updates, alerts, and pipeline progress stream to the operator without polling delay.
- **FR-010a**: System MUST support two-way conversational communication between operators and running agents via a Discord bot (discord.js), where agents behave as chat participants that can send updates, ask questions, request approvals via interactive buttons, and receive commands via slash commands or natural language. The bot runs as a service alongside the orchestrator, routing messages between Discord channels and agents via Redis.
- **FR-011**: System MUST support parallel execution of agents across entirely different domain spaces without interference. Parallel workflows are managed by spawning workflow lead agents that each manage their own hierarchical agent teams under the single root orchestrator.
- **FR-011a**: Parent agents (orchestrator or any agent with sub-agents) MUST be able to resteer running child agents via live directive injection through the message queue. The child agent receives the directive as a priority message, re-evaluates its approach, and adjusts course without restarting — preserving accumulated context and working state. Resteering events MUST be logged in the audit trail.
- **FR-011b**: Inter-agent conflicts (resource contention, budget competition, duplicate work) MUST be resolved via parent-mediated escalation. Agents detect contention and escalate to their shared parent via the message queue. The parent evaluates priorities and context to resolve the conflict. Cross-sub-tree conflicts escalate up the hierarchy to the nearest common ancestor.
- **FR-012**: System MUST automatically detect agent failures (crashes, infinite loops, unresponsiveness) and take corrective action per configurable retry policies.
- **FR-012b**: When agents encounter external service failures (rate-limiting, blocking, outages), the agent MUST autonomously determine its own recovery strategy (backoff, identity/proxy rotation, alternative service substitution) without orchestrator involvement. Operator escalation MUST only occur as a true last resort after the agent has exhausted its autonomous recovery options.
- **FR-012c**: Agents MUST implement self-learning from failures. When an agent resolves an incident (external failure, unexpected behavior, workaround discovery), it MUST persist the resolution to the shared knowledge base so that all agents benefit. Future encounters with similar issues MUST be resolved faster without repeating failed approaches.
- **FR-012d**: The system MUST maintain a shared knowledge base accessible to all agents with high read performance. Learned resolutions, discovered patterns, effective strategies, and operational insights MUST be written to this shared store so the system improves collectively — not in isolation. Every agent spawned MUST benefit from the cumulative knowledge of all prior agents. The system's overall autonomous capability MUST measurably improve over time.
- **FR-012a**: Agent liveness detection MUST use a three-layer approach: (1) heartbeat timeout for hang detection, (2) resource/cost heuristics (CPU, memory, LLM API spend rate) for infinite loop detection, and (3) periodic LLM self-assessment ("are you making progress?") for subtle stall detection. Thresholds and remediation actions (kill, restart, escalate) MUST be configurable per agent.
- **FR-013**: The orchestrator MUST use LLM-powered dynamic goal decomposition to break any high-level goal into sub-tasks, determining agent count, types, capabilities, and sequencing at runtime. No pre-built workflow templates are required. The orchestrator MUST store past decomposition outcomes (successes and failures) in its persistent memory to improve planning over time. Multi-stage workflows (e.g., research, build, deploy, market, reinvest) emerge from this dynamic planning — the self-growing business pipeline is one such emergent workflow.
- **FR-014**: System MUST implement a structured alerting system with configurable alert rules, severity levels (info/warning/critical), and persistent alert history. Alerts MUST be surfaced on the management dashboard as a prioritized, reviewable feed with acknowledgment workflow and smart banner summaries for active critical/warning alerts. Discord notifications (via outbound webhooks) remain the real-time paging mechanism for security events, budget threshold breaches, and actions requiring approval. Alert rules MUST be operator-configurable (e.g., cost velocity thresholds, agent failure rates, security anomalies).
- **FR-015**: System MUST persist agent state and output so that work is not lost on agent or orchestrator restart. Idle agents MUST be suspended to free compute resources, and resumed on demand when new work arrives or their pipeline stage triggers. Agents requiring continuous presence (e.g., monitoring a live application) MUST remain active.
- **FR-015c**: When core infrastructure degrades (secrets vault unavailable, network partitions, credential expiry), agents MUST operate in graceful degradation mode: continue non-sensitive work using cached credentials and state, queue sensitive or infrastructure-dependent operations, and automatically self-heal (replay queued operations) when infrastructure recovers. Agents MUST NOT block entirely on transient infrastructure failures.
- **FR-015b**: System MUST implement durable execution for long-horizon tasks. Agent progress MUST be checkpointed at meaningful intervals as full cognitive snapshots containing: current goal, progress state, decision log (what was tried and why), pending actions, working artifacts, and a structured handoff prompt. Checkpoints MUST survive agent crashes, orchestrator restarts, infrastructure failures, and LLM provider outages. On recovery, agents MUST resume from the last checkpoint with full situational awareness rather than restarting from scratch. Long-horizon work MAY span multiple sequential agent instances, with the orchestrator using checkpoint handoff prompts to maintain continuity of context and reasoning across instances.
- **FR-015a**: When compute resources are exhausted, the orchestrator MUST use a priority queue to suspend lower-priority/idle agents and free resources for higher-priority work. Resource constraints (compute, memory, API budget) MUST be surfaced to the orchestrator as signals that inform its scheduling and growth decisions. The system MUST scale elastically — allocating and releasing compute dynamically based on workload and budget, with no fixed agent ceiling.
- **FR-016**: System MUST support configurable AI provider and model selection per agent, allowing different agents to use different LLM backends (e.g., Claude, GPT, open-source models) based on operator configuration.
- **FR-017**: Agent definitions MUST be structured as a natural language mission/prompt combined with configurable parameters: model, budget, capabilities, and tools. Definitions are created and managed through the management dashboard.
- **FR-018**: Agents MUST be self-aware of their configured capabilities and tool constraints. When an agent determines that its current toolset or model is genuinely insufficient to complete a task, it MUST be able to request capability upgrades (additional tools, stronger model) from the orchestrator. Upgrade requests MUST include justification demonstrating necessity (not speculation) and are subject to operator-configured approval policies. All upgrade requests and outcomes MUST be logged.
- **FR-021**: The orchestrator MUST implement a heartbeat scheduler that wakes at a configurable interval (default: 30 minutes) to perform autonomous monitoring and work advancement. Each heartbeat MUST follow a cheap-checks-first pattern: run lightweight health and status checks (agent liveness, resource usage, pipeline progress, pending approvals) before invoking the LLM. If nothing requires attention, the heartbeat returns immediately without LLM cost. The heartbeat interval and active hours MUST be operator-configurable.
- **FR-021a**: On each heartbeat, the orchestrator MUST evaluate standing orders (operator-defined goals, pipeline advancement opportunities, growth strategies) and take autonomous action where appropriate — spawning new agents, reallocating resources, advancing pipelines, or flagging issues to the operator.
- **FR-020**: Each agent MUST maintain a layered persistent memory system comprising: (1) a durable long-term memory store for learned facts, decisions, preferences, and operational knowledge, and (2) ephemeral session logs capturing running context and activity for each execution session. Memory is the source of truth; the LLM context window is treated as a volatile cache.
- **FR-020a**: Before any context window compaction or session termination, the system MUST trigger a pre-compaction flush — an automatic agentic turn instructing the agent to persist any important in-context information to its durable memory files before that context is lost. Information not flushed before compaction is permanently lost.
- **FR-020b**: On each agent turn or new session, the system MUST perform auto-recall — searching the agent's indexed memory for content relevant to the current task/message and injecting matching context into the agent's working context. This ensures cognitive continuity even after compaction or across session boundaries.
- **FR-020c**: Agent memory files MUST be chunked, embedded, and indexed (using hybrid vector + full-text search) to enable fast semantic retrieval. Agents MUST have explicit tools for memory search (semantic recall) and memory read (targeted file/section access).
- **FR-020g**: The management dashboard MUST surface memory and cognitive system health metrics including: memory write/read rates per agent, retrieval hit quality (average relevance score), shared knowledge base entry count and growth rate, reflection/consolidation frequency, and per-agent memory utilization. These metrics MUST enable the operator to verify the system is learning and improving over time per FR-012d.
- **FR-020f**: Agents MUST perform periodic memory reflection and consolidation at a configurable interval. Reflections MUST produce structured summaries covering: key decisions and rationale, lessons learned, unresolved open loops, and updated mental models. Older granular memory entries MUST be consolidated into higher-level summaries to prevent unbounded growth. Entries that are outdated, contradicted by newer information, or redundant MUST be pruned. The consolidation process MUST preserve all high-importance learnings while reducing storage and retrieval cost.
- **FR-020e**: Memory MUST implement a quality pipeline at both write and read time. At write time: noise filtering MUST reject low-value content (greetings, meta-chatter, trivial observations) before storage, and each accepted entry MUST receive an importance score (0–1) computed by a lightweight LLM call (or heuristic fallback when LLM is unavailable) evaluating novelty, decision significance, and operational impact. At read time: retrieval MUST apply a multi-stage scoring pipeline comprising recency boost (exponential decay favoring newer memories), importance weighting (boosting high-importance entries), time decay (long-term degradation with configurable half-life), MMR diversity (demoting near-duplicate results above a similarity threshold), and a minimum score threshold to discard irrelevant matches. Adaptive retrieval MUST skip queries unlikely to benefit from memory lookup to reduce LLM and embedding costs.
- **FR-020d**: Memory MUST be organized in three strictly isolated tiers: (1) private agent memory — scoped to the owning agent, inaccessible to all other agents per FR-004; (2) shared knowledge base — readable and writable by all agents (FR-012d); (3) orchestrator memory — accessible only to the orchestrator. Agents MUST explicitly publish learnings to the shared knowledge base; private memories are never automatically promoted.
- **FR-022**: Agents MUST be able to author structured, reusable skills — named procedures with defined triggers, sequential steps, inputs, and outputs — that codify learned patterns into executable capabilities. Skills MUST be stored in the shared knowledge base (FR-012d) with versioning. Before a skill is promoted to shared availability, it MUST pass a validation gate: the authoring agent demonstrates at least one successful execution. Any agent MAY invoke a shared skill by name. Skill authorship, invocation, and version history MUST be logged in the audit trail.
- **FR-022b**: Agents MUST proactively discover new tools, APIs, MCPs, and services when their current capabilities are insufficient for a task. Discovery includes searching package registries, API directories, MCP server listings, and documentation. When an agent successfully integrates a discovered tool, it MUST codify the integration as a shared skill (FR-022) — including installation steps, authentication patterns, usage procedures, and failure modes — so all future agents can leverage it without re-discovering. Discovery attempts (successful and failed) MUST be logged.
- **FR-022a**: Skills MUST be structured as: (1) a descriptive name and trigger condition (when to use), (2) typed inputs and outputs, (3) ordered procedural steps, and (4) success/failure criteria. Skills are distinct from knowledge base entries — entries describe information, skills describe executable procedures.
- **FR-022d**: The management dashboard MUST surface skill observability metrics: total skill count and growth rate, invocation frequency (7d/30d), per-skill success/failure rates, top skills ranked by usage, authoring agent attribution, version history timeline, and deprecation events. These metrics MUST enable the operator to verify the skill system is producing measurable value per FR-012d.
- **FR-022c**: Skills MUST implement self-healing lifecycle management. When a skill invocation fails, the invoking agent MUST attempt to diagnose and repair the skill (updating steps, fixing broken integrations, adjusting for API changes) and publish a new version. After a configurable number of consecutive failures across different agents (default: 3), the skill MUST be auto-deprecated, removed from active availability, and flagged for re-authoring. Skill deprecation events MUST be logged and surfaced on the management dashboard.
- **FR-030**: The orchestrator MUST expose a `/health` endpoint that reports the health status of the orchestrator process, Redis connectivity, and PostgreSQL connectivity. The management dashboard MUST surface infrastructure health status alongside agent health. Systemd watchdog MUST restart the orchestrator process on crash or unresponsiveness.
- **FR-029**: On first run (no passkey registered, empty database), the management dashboard MUST present a guided setup wizard that walks the operator through: (1) passkey registration, (2) AI provider API key configuration, (3) payment method setup, (4) Discord webhook configuration, and (5) spawning a test agent to verify end-to-end system health. The wizard MUST NOT allow access to the main dashboard until all critical setup steps are completed.
- **FR-028**: System MUST perform automated daily backups of all PostgreSQL data (agent state, memory, knowledge base, financial ledger, identity registry) with 90-day retention. Audit logs MUST be retained indefinitely. Backups MUST be verified restorable on a configurable schedule (default: weekly). The operator MUST be able to trigger a manual backup and restore from any retained snapshot via the management dashboard.
- **FR-027**: The management dashboard MUST authenticate the operator using Passkey/WebAuthn (passwordless, phishing-resistant). The operator registers a passkey during initial system setup. Unauthenticated access to any dashboard endpoint MUST be denied.
- **FR-026**: Secrets MUST be accessible to agents only through a secure API proxy — never injected as raw environment variables or written to the agent's filesystem in plaintext. The proxy MUST maintain an operator-configurable domain allowlist per secret; outbound requests containing credentials MUST be blocked if the destination domain is not on the allowlist for that secret. All secret access (reads, usage in outbound requests, and blocked attempts) MUST be logged in the audit trail with agent ID, secret identifier (not value), destination domain, and timestamp. Blocked exfiltration attempts MUST trigger a security alert (FR-014).
- **FR-025**: When agents discover and integrate new tools, APIs, MCPs, or services (per FR-022b), the system MUST apply a tiered security gate. Tools from operator-configured trusted sources (known package registries, verified publishers, allowlisted domains) MUST be auto-approved with the integration logged in the audit trail. Tools from unknown or unverified sources MUST require explicit operator approval before the agent can execute them. The operator MUST be able to manage the trusted source allowlist via the management dashboard. All discovery attempts (approved and denied) MUST be logged with source, tool identifier, and verification status.
- **FR-024**: The orchestrator MUST periodically verify the integrity of each running agent's system prompt, mission definition, and behavioral configuration against SHA256 checksums computed at spawn time. If drift is detected (content differs from the checksummed original), the system MUST auto-restore the agent's cognitive configuration to the known-good state, log the drift event as a security incident (agent ID, changed content hash, timestamp), and notify the operator. Integrity checks MUST run at a configurable interval (default: every heartbeat cycle per FR-021).
- **FR-023**: System MUST scan all external content (scraped web pages, API responses, user-generated content, downloaded files) for prompt injection markers before feeding it to an agent's LLM context. Flagged content MUST be quarantined — excluded from the agent's reasoning context — and logged as a security event with the source URL, injection pattern detected, and affected agent ID. Quarantined content MAY be reviewed by the operator via the management dashboard. The scanning layer operates as a pre-processing gate between raw external data and the agent's cognitive pipeline.
- **FR-019**: Agents building revenue-generating applications MUST integrate standard payment processors (e.g., Stripe) under the operator's master payment account. Revenue MUST be automatically routed back into the system's financial ledger, split per FR-008a (default 20% operator / 80% agent reinvestment), and made available for reinvestment allocation. The system MUST maintain a real-time financial ledger tracking all inflows, outflows, splits, and reinvestment balances per agent and per venture.

### Key Entities

- **Orchestrator**: The single root LLM-powered agent that manages the lifecycle of all top-level agents, enforces security policies, and coordinates work across domains and workflows. Runs 24/7 via a heartbeat scheduler (FR-021). For parallel workflows, spawns "workflow lead" agents that each manage their own hierarchical agent teams. Maintains its own persistent layered memory (FR-020) for institutional knowledge — operator preferences, growth strategies, resource allocation patterns, and system-level decisions — distinct from the shared knowledge base.
- **Specialized Agent**: An isolated, autonomous execution unit spawned by the orchestrator or a parent agent to accomplish a specific goal. Has full computer usage capabilities within its sandbox. Powered by a configurable AI provider/model chosen by the operator. Self-aware of its configured capabilities and able to request upgrades when demonstrably insufficient. Can itself spawn and manage sub-agents, acting as a mini-orchestrator for its children (hierarchical agent tree). Budget and permissions are inherited from and constrained by the parent.
- **Agent Definition**: A structured template comprising a natural language mission/prompt and configurable parameters (model, budget, capabilities, tools) that defines how a specialized agent is instantiated. Created and managed through the management dashboard.
- **Agent Environment**: An E2B cloud-native sandbox providing a full virtual desktop in which a specialized agent operates, including its own file system, browser, terminal, screen/mouse/keyboard access, and unrestricted network access. All agents start from a minimal base sandbox and dynamically self-provision any tools they need at runtime. Each sandbox runs a custom Node.js/TypeScript agent runtime process that drives the LLM-call loop, tool execution, memory operations, and orchestrator communication via Redis.
- **Identity**: A digital identity (email, phone number, voice profile, service account) created and managed by an agent, tracked in the identity registry.
- **Secret**: A credential, API key, or payment method stored in the vault and injected into agent environments on a need-to-know basis.
- **Pipeline**: A multi-stage workflow (e.g., research > build > deploy > market > reinvest) that defines the sequence of goals agents pursue autonomously.
- **Audit Log Entry**: An immutable record of an agent action, containing agent ID, action type, timestamp, outcome, and any associated metadata.
- **Skill**: A structured, reusable, executable procedure authored by an agent from learned patterns. Comprises a name, trigger condition, typed inputs/outputs, ordered steps, and success/failure criteria. Stored in the shared knowledge base with version history. Any agent can invoke shared skills by name. Distinct from knowledge entries (passive information) — skills are active, executable capabilities.
- **Shared Knowledge Base**: A system-wide store of learned resolutions, patterns, strategies, operational insights, and authored skills contributed by all agents. Stored in PostgreSQL with pgvector, with structured metadata (tags, categories, resolution type, contributing agent). Indexed with hybrid vector + BM25 full-text search for fast semantic and keyword retrieval. Every agent reads from this store at spawn and during execution, and writes back learnings on resolution of novel problems. The collective intelligence of the system grows continuously.
- **Agent Memory**: A layered persistent memory system per agent backed by PostgreSQL with pgvector, consisting of durable long-term memory (learned facts, decisions, preferences) and ephemeral session logs (running context per execution session). Indexed for hybrid semantic + full-text retrieval. The source of truth for agent cognition — the LLM context window is a volatile cache over this store.
- **Approval Gate**: A configurable checkpoint where agent actions are paused pending operator approval before proceeding.

- This is a single-operator, bespoke personal system. There is no multi-tenancy, user management, or shared access. Authentication is single-user only.
- The orchestrator runs on a single cloud VM (EC2, GCE, or DigitalOcean droplet) managed by systemd for process supervision and auto-restart.
- The operator has access to a cloud provider (e.g., AWS, GCP, Azure) with sufficient compute, storage, and networking resources. PostgreSQL with pgvector runs self-hosted in Docker Compose on the same VM as the orchestrator.
- The operator provides initial funding/payment methods for agent activities (service subscriptions, domain purchases, hosting costs).
- External services (email providers, phone number services, hosting platforms) allow automated account creation within their terms of service.
- The operator accepts that fully autonomous agent actions carry inherent risk and configures appropriate guardrails and approval gates.
- The orchestrator is a Node.js/TypeScript long-running process, chosen for best-in-class AI SDK ecosystem and E2B SDK support. LLM integration uses the Vercel AI SDK for unified multi-provider support.
- E2B cloud-native sandboxes provide agent isolation with full virtual desktop capabilities; hardware-level isolation is not required.
- The management dashboard is built with Next.js + React, sharing TypeScript with the orchestrator backend.
- The system uses industry-standard encryption (TLS 1.3 for transit, AES-256 for rest) for all security requirements.
- Agent computer usage is achieved through E2B cloud-native sandboxes providing full virtual desktops with browser, terminal, file system, and screen/mouse/keyboard access via API.
- The system is deployed via Docker Compose on the single cloud VM, bundling all services (orchestrator, Redis, PostgreSQL, dashboard) into a reproducible stack.
